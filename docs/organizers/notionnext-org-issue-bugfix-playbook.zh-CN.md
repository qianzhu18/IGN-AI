# NotionNext 组织者 Issue 修复与社区范式升级 Playbook

记录日期：2026-05-14  
适用范围：`notionnext-org/NotionNext` 上游协作 + `IGNAI` 本地社区官网二开  
角色视角：组织者 / 共创者 / 持续贡献者

## 1. 这份文档解决什么问题

这份文档不是单纯的 issue 清单，也不是单纯的产品愿景说明。

它要同时服务 4 件事：

1. 帮助持续刷有效贡献，而不是零散修小点
2. 帮助把容易合并的 bugfix 和更长期的社区能力升级拆成两条线
3. 帮助把本地二开实践沉淀为可上游贡献的通用能力
4. 帮助后续写复盘、写简历、写开源协作叙事时有稳定素材

一句话概括：

```text
先用高成功率 bugfix 建立连续贡献记录，再用真实社区场景推动 NotionNext 从个人博客框架向社区官网范式演进。
```

## 2. 当前协作判断

截至 `2026-05-14`，当前上游公开信号已经很明确：

- `NotionNext` 约 `11.4k stars / 14.6k forks`
- 社区官网方向 issue 已建立：`#4035`
- 配套 roadmap 文档 PR 已合并：`#4036`
- 本账号公开可查已有多条已合并 PR
- 本地仓库已完成 `Member` 基础路由、作者关联、申请池雏形等验证

所以现在最合理的工作模式不是“继续零散改站”，而是双轨并行：

### 轨道 A：上游 bugfix 贡献线

目标：

- 选择容易复现、范围清晰、容易合并的 issue
- 稳定输出 PR 记录
- 建立“能修核心问题”的维护者形象

### 轨道 B：社区范式升级线

目标：

- 在本地仓库继续把 `Member / Event / Join / Ops` 路线做实
- 把“社区官网”从产品叙事变成已验证实现
- 再把其中可复用的底层能力拆回上游

## 3. 总原则

### 原则 1：优先修“容易复现 + 容易验收 + 影响多人”的问题

优先选择：

- 明确是 regression 的问题
- issue 描述完整、有复现线索
- 改动范围集中在数据处理 / 解析 / 路由 / 输出
- 可以补测试

暂缓选择：

- 依赖特定平台环境且复现条件复杂
- 需要大幅重构主题层
- 讨论多但问题边界不清

### 原则 2：先拿下 bugfix 贡献密度，再推进范式升级话语权

如果只谈方向，不补落地修复，容易被看成“提想法的人”。  
如果只修 bug，不建立产品判断，又很难把自己从普通贡献者抬到“方向共创者”。

正确节奏：

```text
bugfix 连续命中 -> 建立信任 -> 推进社区能力底座 -> 拆上游 PR -> 形成个人标签
```

### 原则 3：上游和本地必须主动拆层

#### 上游优先保留

- Notion 数据兼容
- page type / route / selector
- 作者与成员关联辅助逻辑
- 通用 schema 文档
- event / registration 通用字段约定

#### 本地优先保留

- `ignai` 品牌视觉
- 社区叙事文案
- Join / Ops 的具体 UI
- 本地运营流程

## 4. 当前建议的 Issue 分层

以下排序基于 `2026-05-14` 公开 issues 快照，目标不是“全做”，而是优先选高胜率项。

## 4.1 第一梯队：优先出手的高胜率问题

这些问题最适合作为持续贡献主力。

### 候选 1：数据库筛选 / page 筛选失效

- `#4039` notion 数据库的筛选显示不生效
- `#3940` page 页面展示筛选后无效

为什么优先：

- 两个 issue 症状接近，可能是同一根因
- 用户影响面广
- 回归意味明显
- 一旦修好，容易补回归测试

建议动作：

1. 先复现筛选逻辑失效
2. 定位数据层是忽略了 Notion 视图筛选，还是站点层做了全量回填
3. 优先争取做成一个通用修复，而不是主题修补
4. 如果根因一致，可考虑一组 issue 对应一个 PR

预期价值：

- 对外是稳定性修复
- 对内能进一步理解 Notion 数据层
- 对社区官网方向也有帮助，因为未来 `Member / Event` 也依赖列表筛选

### 候选 2：Apple Music 单曲嵌入识别错误

- `#3914` 嵌入 Apple Music 单曲播放器时会被识别成视频

为什么优先：

- 复现条件相对清楚
- 问题边界集中在 embed URL 识别
- 潜在改动点局部
- 容易单独成 PR

建议动作：

1. 先确认当前 embed 识别逻辑如何判断 music / video
2. 对 `https://embed.music.apple.com/.../song/...` 做格式兼容
3. 如果 Notion 会自动转旧格式，也可在渲染前做 normalize

预期价值：

- 小而完整
- 合并概率高
- 适合作为穿插型贡献

### 候选 3：RSS 相关残留问题的收口

- `#3983` RSS 订阅 404
- `#3931` RSS 失效 / 502

为什么优先：

- 你已经有 RSS 相关合并记录
- 这条线已经形成个人上下文
- 修复时更容易获得维护者信任

建议动作：

1. 先核对已合并 PR 是否只修了一部分路径
2. 判断问题是路由路径、构建产物生成，还是部署环境差异
3. 如果不是新 bug，而是文档 / 默认配置误导，也可以先补文档说明

预期价值：

- 复用你已有贡献带来的信用
- 能继续巩固“构建与输出稳定性”标签

## 4.2 第二梯队：中等难度，适合在第一梯队稳定后推进

### `#3836` 公式区块不能正常显示

判断：

- 用户价值高
- 但可能牵涉渲染器、样式、Math 支持或主题表现
- 复现与修复半径比第一梯队大

建议：

- 适合做成后续专项，不建议现在拿来刷短周期贡献

### `#3890` Notion API 429 / 构建超频

判断：

- 价值很高
- 但属于核心稳定性问题
- 涉及构建并发、请求节流、数据抓取策略

建议：

- 这是“抬地位”的问题，不是“刷贡献速度”的问题
- 可以先记为中期攻坚项，待前面连续合并几条后再打

## 4.3 第三梯队：战略型路线，不以短期 issue 数量为目标

## 6. 每次挑 issue 的判断模板

新 issue 到手时，先用这 6 个问题过一遍：

1. 能不能在 30 分钟内初步定位入口文件？
2. 能不能在 1 小时内写出复现步骤？
3. 能不能在一个小 PR 里收束？
4. 能不能补测试或至少补稳定验收步骤？
5. 修完后是不是对多数用户有明显价值？
6. 这件事会不会反过来帮助 `Member / Event / Ops` 主线？

如果 `1~4` 大部分答案都是“否”，那就先不要把它作为刷贡献主项。

## 7. PR 拆分规则

### 适合单独 PR 的内容

- 一个明确 bug 的修复
- 一个数据兼容点
- 一个输出路径问题
- 一个 embed / parser 兼容点

### 不适合混在一起的内容

- bugfix + 主题美化
- bugfix + 大量重构
- 上游通用逻辑 + IGNAI 本地 UI
- 文档路线 + 功能实现 + 大量页面改版

## 12. 并行刷贡献 TODO 列表

这个列表专门服务一种工作方式：

```text
我在当前对话改一段，
你开一个新对话改另一段，
尽量把主流、容易改、真有效的问题并行刷完。
```

执行原则：

1. 每个对话只认领一个明确块
2. 尽量避免两个对话同时改同一组文件
3. 先做高胜率 bugfix，再做中期专项，再做社区范式升级
4. 每做完一块，就补测试 / 验收说明 / issue 叙事

### 一轮刷完后的完成标准

这一轮不以“做了很多”为标准，而以这 4 件事为标准：

- 至少 2 组 issue 有明确定位结果
- 至少 1 个 bugfix 形成干净最小 diff
- 至少 1 条线补上回归测试
- 每条线都能说清楚：
  - 改了什么
  - 为什么值得合并
  - 哪些属于上游
  - 哪些留在本地

## 13. 最后的工作判断

这条线现在最重要的，不是“多做一些功能”，而是形成一种稳定范式：

```text
上游用 bugfix 建立贡献密度，
本地用社区产品验证新范式，
再把验证过的通用层回流到上游。
```

只要这个节奏跑通，你这条二开路线就不是普通魔改，而是在真正参与一个开源项目的下一阶段定义。
