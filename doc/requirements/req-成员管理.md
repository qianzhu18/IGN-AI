# IGNAI 成员管理需求评估

记录日期：2026-05-03

## 当前结论

暂时不立即开发成员管理功能，先进入 TODO 和排期。

如果后续希望把成员管理中的“公开展示部分”反向沉淀到 NotionNext 上游，请同时参考：

```text
doc/requirements/req-NotionNext社区能力提案.md
```

这份文档会把：

```text
成员头像预览
bio 展示
成员目录
活动与内容分流
```

拆成更适合提 issue / PR 的上游能力，而不是直接把完整社区运营系统整体上推。

当前项目已经接入 Sanity Studio，用于活动和现场记录编辑。Supabase 目前还没有真正接入生产数据，只保留了加入申请写入的服务端适配代码；只有配置 `SUPABASE_URL` 和 `SUPABASE_SERVICE_ROLE_KEY` 后才会启用。

成员管理第一阶段建议先用 Sanity 做公开成员资料编辑，不急着上 Supabase。

## 为什么先不用 Supabase

成员预览页的核心数据很简单：

```text
头像
昵称 / 姓名
身份标签
城市 / 地区
一句话简介
兴趣标签
个人链接
是否首页展示
排序
```

这些更像内容数据，而不是强业务数据。Sanity 更适合非技术成员编辑、上传头像、调整展示顺序和发布内容。

Supabase 更适合后续这些场景：

```text
成员自己登录修改资料
加入申请自动转成员
成员权限 / 管理员权限
私密联系方式
报名记录和成员身份关联
活动签到
成员状态流转
```

所以建议路线是：

```text
Sanity 先做公开成员资料
Supabase 后做身份、权限和业务记录
```

## 目标体验

参考 2050 成员列表的轻量信息密度，但视觉上保持 IGNAI 官网风格。

用户可以在官网看到：

```text
首页成员预览区：展示 6-8 位精选成员
/members 页面：展示完整成员列表
/members/[slug] 页面：展示成员详情
```

后台编辑者可以在 `/studio` 里：

```text
新建成员
上传头像
填写昵称、身份、城市和简介
配置标签与链接
设置是否首页精选
调整排序
发布或隐藏成员
```

## 建议字段

### 基础信息

```text
displayName：公开显示名，必填
slug：URL 标识，必填
avatar：头像
title：身份标题，例如 AI Builder、产品经理、创作者
city：城市 / 地区
bio：短简介，建议 80 字以内
longBio：详情页简介，可选
```

### 社区信息

```text
role：成员角色，例如 Builder、Creator、Connector、Researcher
tags：兴趣标签，例如 Agent、Product、Content、Startup
joinedAtText：加入时间文案，例如 2026 / 05
isFeatured：首页精选
sortOrder：排序
status：展示状态
```

### 外部链接

```text
website
x / twitter
github
xiaohongshu
wechatQr 可选
```

微信、手机号、邮箱等联系方式默认不公开，除非成员明确授权。

## 页面方案

### 首页成员预览

位置建议放在 Field Notes 和 Join 之间。

```text
COMMUNITY MEMBERS
一群正在把 AI 想法做成真实行动的人。
```

展示方式：

```text
6 张成员卡
头像 + 名称 + 角色 + 城市 + 一句话简介 + 标签
CTA：查看全部成员
```

### 成员列表页

路径：

```text
/members
```

功能：

```text
搜索昵称 / 城市 / 标签
按角色筛选
按最新加入或排序展示
移动端单列卡片
桌面端 2-3 列卡片
```

### 成员详情页

路径：

```text
/members/[slug]
```

内容：

```text
头像
名称
角色
城市
简介
标签
公开链接
参与过的活动 / 记录，后续再关联
```

## 隐私与审核

成员资料属于个人信息，必须设置最小公开原则：

```text
只公开成员授权展示的字段
联系方式默认不公开
头像、真实姓名、城市需要成员同意
后台保留 draft 状态，发布前人工确认
允许成员要求下架或修改
```

不建议第一版做：

```text
公开手机号 / 邮箱 / 微信号
成员自由上传后自动公开
未授权搬运头像和简介
复杂社交关系图
私信系统
```

## 数据方案评估

### 方案 A：Sanity-only

适合第一版。

优点：

```text
开发快
编辑体验已经接入
头像上传方便
适合公开内容
Vercel 部署简单
```

限制：

```text
没有成员自助登录
没有复杂权限
私密字段不适合放太多
```

### 方案 B：Supabase-only

不建议作为第一版。

优点：

```text
适合成员账号和权限
适合业务表关联
适合申请、报名、签到
```

限制：

```text
需要 Auth、RLS、Storage、后台表单
非技术编辑体验不如 Sanity
开发复杂度明显升高
```

### 方案 C：Sanity + Supabase

适合第二阶段。

```text
Sanity 管公开展示内容
Supabase 管成员身份、申请、报名和私密信息
```

这是长期最稳的方案。
