# 成员资料上传与前端对标方案

记录日期：2026-06-03
状态：调研完成，等待确认后执行

## 1. 这次校准后的判断

这轮不继续追求“更多漂亮网站”，而是收束成两个真实可落地的目标：

1. P0：先补齐成员资料采集、头像上传、图床 URL 回写 Notion 的运营闭环。
2. P1：再按 Rig AI + The State of AI at Work 的方向重构前端叙事，而不是继续堆卡片。

不采用的方向：

- Ausdata AI：中文站整体观感和 IGNAI 需求不匹配，不作为对标。
- MotionSites：适合做灵感库，不适合作为当前官网复刻目标，驾驭成本偏高。
- PageCoder.ai：极简方向不足以解决 IGNAI 当前“组织表达弱”的问题，暂不采用。

## 2. P0：成员资料上传与 Notion 回写

### 2.1 为什么排 P0

当前 Member 前端已经能展示 `avatar`、`bio`、`role`、`quote`、社交链接等字段，但头像和简介仍依赖手动录入。继续打磨成员目录 UI 前，必须先让真实成员资料进入 Notion 后端，否则前端会一直被演示数据和缺头像状态拖住。

### 2.2 目标链路

```text
成员/管理员提交资料
-> 上传头像
-> 压缩 / 裁切 / 校验
-> 上传到 S3 兼容图床 / OSS
-> 得到公网 URL
-> 写入 Notion Member.avatar
-> 同步写入 bio / role / website / social fields
-> 管理员审核后 status = Published
```

### 2.3 需要支持的字段

沿用现有契约，不新增无关字段：

- `title`
- `slug`
- `role`
- `bio`
- `summary`
- `avatar`
- `quote`
- `featured`
- `verified`
- `sortOrder`
- `author_slug`
- `social_github`
- `social_x`
- `social_linkedin`
- `website`

参考现有文档：

- `docs/member-schema.md`
- `docs/member-profile-collection-template.zh-CN.md`

### 2.4 可复用的 OSS / 图床线索

已在 Obsidian 写作发布流程中找到可复用思路：

```text
/Users/mac/qianzhu Vault/qianzhu Wiki/.obsidian/plugins/obsidian-to-mp/main.js
```

相关能力：

- `CloudImageUploader.uploadBlob(blob, fileName, { verifyPublicRead })`
- 支持 S3 兼容 Endpoint / Bucket / Region
- 支持 AccessKey ID / Secret Access Key
- 支持 `Public Base URL`
- 支持 `Path Prefix`
- 上传后可校验公网可读
- 阿里云 OSS 推荐 `virtual-hosted-style`
- 如果云端图床不可用，公众号流程会 fallback 到微信上传；官网成员系统不需要这个 fallback

### 2.5 官网实现建议

不要直接复制 Obsidian 插件的打包产物，应抽象为服务端 API：

```text
pages/api/admin/member-avatar-upload.ts
pages/api/admin/member-profile-update.ts
lib/server/cloudImageUploader.ts
lib/server/notionMemberUpdater.ts
```

第一阶段建议只允许管理员使用，避免权限系统未完成时暴露写 Notion 能力：

1. 管理员选择图片和成员。
2. 服务端压缩并上传图床。
3. 服务端写入 Notion `avatar` URL。
4. 可同步更新简介、角色、链接字段。

第二阶段再做成员自助提交：

1. 成员提交资料草稿。
2. 资料进入待审核状态。
3. 管理员确认后写入 Notion / 发布。

## 3. 前端对标方向

### 3.1 主对标一：Rig AI

参考：

- https://rig.ai/
- https://www.awwwards.com/sites/rig-ai

可借鉴点：

- 大型产品宣言首屏：一句话明确主张，不靠卡片解释。
- 强橙色 + 黑色断层：和 IGNAI 当前 heat / dark 体系贴合。
- 问题区块先行：先讲“当前 AI 学习/社区/产品探索的问题”，再讲 IGNAI 怎么解决。
- 横向 marquee 信号条：可承载 `Builders / Events / Field Notes / Members / Open Source`。
- 阶段化叙事：Problem -> Introducing -> Approach -> Capabilities -> CTA。

落到 IGNAI：

```text
Hero:
长沙 AI Builders 社区
From scattered experiments to shared momentum.

Problem:
AI 学习很热，但真实协作、线下连接、项目沉淀很分散。

Introducing IGNAI:
把成员、活动、文章、开源贡献串成一个可持续的社区系统。

Capabilities:
Member profiles / Events / Field notes / Open source contribution / Local network
```

### 3.2 主对标二：The State of AI at Work

参考：

- https://asana.com/resources/state-of-ai-work?chapter=introduction
- https://www.awwwards.com/sites/the-state-of-ai-at-work

可借鉴点：

- 报告型长页，不是普通落地页。
- 章节 tab 固定在首屏下方，天然适合“社区报告 / 活动复盘 / AI 趋势观察”。
- 网格底纹 + 大字号标题，信息感强但不浮夸。
- 用数字讲故事，而不是靠装饰堆视觉。

落到 IGNAI：

```text
Community Report:
Chapter 1: Why IGNAI
Chapter 2: Members
Chapter 3: Events
Chapter 4: Field Notes
Chapter 5: Open Source

每章只做一个核心表达：
成员数量 / 活动节奏 / 内容产出 / 开源贡献 / 下一步方向
```

### 3.3 融合原则

Rig AI 负责“态度和冲击力”，Asana 负责“结构和可信度”。

不要做：

- 全站卡片墙
- 过多 3D / WebGL 复杂场景
- 纯灵感库式 prompt gallery
- 只好看但不服务成员数据的视觉组件

要做：

- 首页是社区叙事长页
- 成员系统是可运营数据
- 成员目录不是卡片墙，而是组织地图 / 人员索引
- 活动和记录区块有报告感与章节感
