# 3003 复刻 3004 执行文档

> 项目：IGNAI 官网  
> 日期：2026-05-07  
> 目标：让 `http://localhost:3003/` 的 NotionNext 项目，在保留 NotionNext 内容与路由能力的前提下，前端 UI 尽可能完整复刻 `http://localhost:3004/` 当前旧版官网界面

---

## 0. 结论先行

可以复刻，但不能靠“继续在现有 3003 主题上补样式”完成。

根本原因不是颜色、图片或组件不够像，而是：

1. `3004` 是一个独立官网结构，首页是 section-driven 的完整叙事页面。
2. `3003` 当前虽然已经叫 `ignai` 主题，但骨架仍然是 NotionNext 主题页思路，首页本质还是“主题首页 + 内容列表”。
3. 两边的数据模型、路由机制、样式体系、动画系统、组件组织方式都不一致。
4. 当前工作树里 `themes/ignai/*` 还处于大量删除/替换混杂状态，继续直接改容易把缓存里的“看起来还能跑”和真实文件状态混在一起。

所以，正确路线不是“继续调主题”，而是：

- 把 NotionNext 降级为内容引擎与路由宿主
- 把 3004 的页面结构当成唯一 UI 基线
- 在 `themes/ignai` 中重建一个“v1 shell”
- 再把 NotionNext 的文章、活动、归档、详情页逐步挂接到这个 shell 上

一句话说，就是：

**不是给 NotionNext 换皮，而是把 3004 的前台壳搬进 NotionNext。**

## 2. 当前状态总结

### 2.1 3004 是什么

`3004` 对应的是引入 NotionNext 之前的旧版官网结构，技术上是独立构建的官网：

- 首页是固定 section 顺序
- 强依赖品牌图像、版式节奏、动效层次
- 不是“文章列表首页”，而是“社区官网首页”
- 文案、图片、CTA、区块编排都服务于统一叙事

其首页主结构可概括为：

1. Hero
2. What is IGNAI
3. Culture
4. Upcoming Events
5. Field Notes
6. Community Roles
7. Join
8. Footer

### 2.2 3003 现在是什么

`3003` 是 NotionNext 分支上的 `ignai` 主题，虽然已经开始做 IGNAI 品牌适配，但当前首页骨架仍来自主题页逻辑：

1. Header
2. Hero
3. Brand ticker
4. Features
5. Team/About
6. Blog/List
7. Announcement
8. CTA
9. Footer

这意味着 3003 当前更像：

- “一个被 IGNAI 品牌改造过的主题首页”

而不是：

- “3004 官网被搬进 NotionNext”

### 2.3 关键现象

当前仓库里同时存在两套与首页相关的前端资产：

1. `themes/ignai/*`
   - 这是当前 3003 真正用于渲染的 NotionNext 主题目录
   - 但其首页结构仍然沿用主题式组织

2. `src/components/sections/*`、`src/content/*`
   - 这里保留了大量 3004 旧官网 section 资产
   - 包括 `HeroSection`、`WhatIsSection`、`CultureSection`、`UpcomingEventsSection` 等
   - 这些才更接近我们真正要复刻的目标结构

因此，复刻不是“从零开始”，而是“把已经存在但尚未接到渲染主干上的旧版 section 体系，正式接管 3003 首页”。

## 3. 为什么 3003 现在不能直接复刻 3004

这里的“不能直接复刻”，不是说技术上做不到，而是说不能通过当前的低层级改法达成。

### 3.1 首页骨架不一致

这是最大原因。

`3004` 的首页是 section-driven 的页面编排，强调：

- 固定叙事顺序
- 左右栏版式关系
- 图文区块密度
- 跨 section 的视觉连续性

`3003` 当前 `themes/ignai/index.js` 仍然是主题骨架：

- Hero
- 品牌滚动
- 特性卡
- 团队区
- 列表区
- CTA

也就是说：

- `3004` 是“品牌官网结构”
- `3003` 是“内容主题结构”

只在现有 `themes/ignai` 上补 CSS，会得到一个“更像 IGNAI 的主题页”，但不会得到“3004 的官网结构”。

### 3.2 数据模型不一致

`3004` 的首页区块使用的是明确的站点级数据结构：

- `siteContent`
- `whatIsContent`
- `cultureContent`
- `events`
- `records`
- `joinContent`

这些数据是围绕固定区块写的。

而 `3003` 当前首页更依赖 NotionNext 的通用数据流：

- `allNavPages`
- `latestPosts`
- `notice`
- `posts`

这套数据天生更适合：

- 博客首页
- 归档页
- 列表主题

不适合直接表达 3004 这种“活动区”“现场记录区”“角色区”“品牌区”的定制首页。

### 3.3 路由与渲染宿主不同

`3004` 使用独立官网页面组织方式。

`3003` 使用 NotionNext Pages Router + 主题布局注入机制：

- `pages/_app.js`
- `pages/index.js`
- `themes/theme.js`
- `themes/ignai/index.js`

这意味着 3003 的首页要通过主题入口接管，而不是像 3004 一样直接由 `app/page.tsx` 作为唯一页面宿主。

所以不能简单把 3004 的页面文件原样贴过来，必须重组到 NotionNext 的主题生命周期中。

### 3.4 样式体系不同

`3004` 那套 section 资产明显带有另一套风格来源：

- 更接近 Tailwind + 组件式 class 体系
- 大量依赖 `max-w-[ch]`、grid、细粒度 spacing 和独立 section class
- 使用 Framer Motion 风格的进入动画组织

`3003` 当前 `themes/ignai` 的主样式宿主是：

- `styled-jsx`
- 主题内部 class 命名
- 原本面向 NotionNext 主题组件的样式分发方式

这不构成硬阻塞，但意味着：

- 不能只做 class 名映射
- 需要决定“旧 section 样式是迁入 theme 样式层，还是保留为局部组件样式”

### 3.5 主题解析优先级曾经有偏差

当前分支里还存在一个已经暴露出来的实际问题：

- Notion 配置中的 `THEME` 可能覆盖 `blog.config.js`
- 导致页面不是渲染 `ignai`，而是落到其他主题

这个问题已经开始修正，但它说明一件事：

**如果不先把 3003 的主题主权稳定住，后续任何“复刻效果”都可能只是缓存或局部错觉。**

### 3.6 当前工作树不稳定

当前 `git status` 里，`themes/ignai/*` 出现了大量：

- 已跟踪文件删除
- 新增文件未纳入统一结构
- 旧组件与新组件混杂

这说明现在不是一个适合“边试边缝”的状态。

如果不先清理/恢复到一个稳定基线，开发会出现三个风险：

1. dev server 用缓存继续显示旧画面
2. 文件系统真实状态与浏览器所见不一致
3. 一旦重启，3003 可能直接失真或报错

## 4. 复刻目标定义

### 4.1 本次复刻不是做什么

不是：

- 仅仅品牌化 NotionNext
- 仅仅把 3004 的图片搬到 3003
- 仅仅让 Hero 像一点
- 仅仅“视觉接近”

### 4.2 本次复刻要做到什么

目标是：

**在 NotionNext 框架内，让 3003 的前台首页、关键内容页、品牌节奏、主要视觉层级尽可能还原 3004 的结构与观感。**

最低标准：

1. 首页 section 顺序与 3004 一致
2. Hero 结构与 3004 一致
3. 主体视觉节奏与 3004 一致
4. 活动区、记录区、Join 区结构与 3004 一致
5. 页面第一屏一眼看上去应被识别为“同一个站”

允许不同之处：

1. 数据来源改为 NotionNext/Notion
2. 部分区块在数据为空时使用 fallback 内容
3. 详情页、归档页仍需符合 NotionNext 的内容渲染机制

## 5. 推荐技术路径

### 5.1 总体策略

采用“双层架构”：

#### 层 A：NotionNext 继续负责

- 路由
- 页面生命周期
- Notion 数据获取
- 文章详情渲染
- 搜索、归档、分类、标签

#### 层 B：IGNAI v1 shell 接管前台结构

- 首页 section 编排
- 站点品牌头图
- 固定区块叙事
- 活动/记录/Join 的官网式呈现
- 页面背景、网格、噪点、动效语义

这样才能让 3003 既保留 NotionNext 的内容后端能力，又真正长成 3004 的前台样子。

### 5.2 首页应该如何重构

不要继续让 `themes/ignai/LayoutIndex` 渲染：

- Brand
- Features
- Team
- Blog

应改为渲染一个新的首页壳，例如：

`HomeShell ->`

1. `HeroSection`
2. `WhatIsSection`
3. `CultureSection`
4. `UpcomingEventsSection`
5. `FieldNotesSection`
6. `CommunityRolesSection`
7. `JoinSection`
8. `FooterSection`

其中：

- section 组件优先复用当前仓库里已经存在的 `src/components/sections/*`
- 这些 section 通过适配层进入 NotionNext 主题，而不是继续挂在独立 `app/page.tsx` 思路上

### 5.3 数据适配层

需要新增一个“首页专用 ViewModel 层”。

原因是：

- NotionNext 给的是通用内容数据
- 3004 section 需要的是定制展示数据

推荐增加一层映射：

`NotionNext raw data -> IGNAI home view model -> section components`

建议最少包含：

- `homeHero`
- `homeEvents`
- `homeRecords`
- `homeCommunity`
- `homeJoin`

数据来源优先级建议：

1. Notion 数据
2. 本地 `src/content/*` fallback
3. 硬编码兜底文案

这样能保证：

- 先完成 UI 复刻
- 再逐步切换为 Notion 可编辑数据

### 5.4 主题入口重组

需要把 `themes/ignai/index.js` 从“改过的 proxio 主题”升级成“IGNAI 主站主题入口”。

建议结构：

1. `LayoutBase`
   - 全局背景层
   - 顶部导航
   - 暗色模式宿主
   - 页面级 footer 容器

2. `LayoutIndex`
   - 不再输出主题式 Blog/Brand/Features
   - 改为直接输出 section-driven 首页

3. `LayoutSlug`
   - 先保留 NotionPage 内容渲染
   - 再逐步套入 IGNAI 风格详情页头部与内容壳

4. `LayoutArchive` / `LayoutPostList`
   - 第二阶段再与首页风格统一

### 5.5 样式策略

不建议本轮去做“大规模样式体系迁移”。

不建议：

- 把整站强行迁回 Tailwind-only
- 把所有 section 改写为另一套 class 方案

建议：

1. 以 `themes/ignai/style.js` 作为全局主题壳样式入口
2. 保留必要的局部组件 class
3. 把 3004 中真正决定观感的设计 token 抽回主题层

优先抽取的内容：

- 色彩变量
- section spacing
- display title 体系
- surface card 体系
- Hero grid
- open grid
- CTA 按钮
- 背景层与网格

### 5.6 动画策略

首页应尽量跟 3004 一致，优先恢复：

- Hero 进入节奏
- section reveal
- card stagger
- 背景缓动层

如果当前主题仍残留 WOW.js / Animate.css 逻辑，建议逐步退场，统一为：

- Framer Motion 或等价统一动画方案

原则：

- 首页 section 动画统一
- 列表页和详情页可暂时保持轻量

## 6. 实施框架

### Phase 0：稳定基线

目标：让 3003 的当前开发底座变得可信。

任务：

1. 恢复 `themes/ignai/*` 到当前分支已提交的稳定状态
2. 清理 dev cache 幻觉
3. 确认 `3003` 实际渲染的是 `ignai` 主题
4. 固定对照入口：
   - `3003` 当前改造版
   - `3004` 旧版基准

产出：

- 一个可以重启、可重复渲染、无缓存误导的基线环境

### Phase 1：首页结构接管

目标：把 3004 的首页骨架正式搬入 3003。

任务：

1. 在 `themes/ignai` 中引入/重建首页 section shell
2. 用 section-driven 首页替换当前主题式首页
3. 先接入本地 `src/content/*` 与 `src/content/events.ts`、`src/content/records.ts`
4. 让首页结构、区块顺序、CTA 位置与 3004 对齐

产出：

- 3003 首页结构与 3004 基本一致

### Phase 2：视觉与动效对齐

目标：让第一眼观感接近 3004，而不是只有结构相似。

任务：

1. 对齐 Hero 双栏结构
2. 对齐 section spacing 与 typography
3. 对齐图片展示方式
4. 恢复背景层、网格、噪点、光晕
5. 恢复主要 reveal 动画

产出：

- 3003 首屏和主体区块的品牌气质基本贴近 3004

### Phase 3：内容页挂接

目标：让 NotionNext 内容能力被整合进新壳。

任务：

1. 活动列表映射到 `UpcomingEventsSection`
2. 记录列表映射到 `FieldNotesSection`
3. 文章详情页套壳
4. 归档/列表页统一视觉

产出：

- 不是只有首页像，而是“整站方向一致”

### Phase 4：Notion 数据正式接管

目标：从本地 fallback 数据平稳过渡到 Notion 数据。

任务：

1. 建立首页 ViewModel 映射
2. 让 section 数据优先读取 Notion
3. 数据缺省时回退本地内容
4. 验证线上可编辑性与本地视觉不冲突

产出：

- 3003 同时拥有 3004 的外观和 NotionNext 的可维护性

## 7. 为什么这条路线是对的

因为它同时满足三件事：

### 7.1 满足你的真实目标

你的目标不是“主题适配”，而是：

- 在 NotionNext 上继续用你自己的站

这条路线承认：

- IGNAI 的前台结构是主角
- NotionNext 只是底层能力，不是前台审美来源

### 7.2 不浪费现有资产

当前仓库里已经有三类资产：

1. 3004 的旧 UI 结构经验
2. 当前 `src/components/sections/*` 的复刻雏形
3. NotionNext 的内容/路由体系

这条路线不是推翻重来，而是把这三类资产重新排位：

- UI 结构优先
- 内容引擎其次
- 主题框架退后

### 7.3 工程风险最小

如果继续沿当前“改主题”的方式做，会不断遇到：

- 越改越像另一个主题
- 结构越来越缝
- 首页和内容页之间调性断裂

而“v1 shell 接管”虽然前期重构重一点，但中后期更稳定，返工更少。

## 8. 主要风险

### 风险 1：当前工作树脏状态导致实现过程不稳定

表现：

- `themes/ignai/*` 有大量删除与新增并存
- 缓存中的页面与真实文件可能不一致

应对：

- 先做 Phase 0，恢复稳定基线，再开工

### 风险 2：首页复刻后，文章页与归档页仍旧“像主题站”

表现：

- 首页像 3004
- 详情页一跳回去，仍是明显的 NotionNext 风格

应对：

- 把列表页/详情页纳入 Phase 3，不只做首页

### 风险 3：Notion 数据结构不够支撑首页定制

表现：

- 活动/记录字段不全
- 首页需要的 badge、subtitle、cover、排序语义不足

应对：

- 先允许本地 fallback 驱动 UI
- 再设计 Notion 字段映射

## 9. 验收标准

当满足以下条件时，可认为 3003 已基本完成对 3004 的复刻：

1. `3003` 首页 section 顺序与 `3004` 一致
2. Hero 的图文结构、按钮布局、信号卡节奏与 `3004` 一致
3. Events / Field Notes / Join 三个核心区块与 `3004` 的结构和视觉意图一致
4. 第一屏截图并排对比时，能明确识别为同一站点的两个架构版本
5. `3003` 保留 NotionNext 的内容获取与详情页能力
6. 页面刷新、重启 dev server 后，渲染结果稳定，不依赖缓存幻觉

---

## 10. 建议的实际开工顺序

建议按下面顺序执行，而不是同步乱改：

1. 先稳定 `themes/ignai` 基线
2. 先让 `LayoutIndex` 接管旧版首页结构
3. 先做首页，不急着碰详情页
4. 首页结构完成后，再做视觉细节
5. 首页稳定后，再把列表页和详情页统一进去
6. 最后才做 Notion 字段映射与内容接管
